# 관리자 백스테이지 레이아웃 디자인 spec

## 상태
- 사용자 승인 완료
- 기준 방향: `백스테이지 투-패턴`

## 목적
- 관리자 페이지를 블로그 메인페이지와 글 상세 페이지의 시각 문법 안으로 다시 정렬한다.
- 운영 화면이 별도 제품처럼 보이지 않게 하면서, 관리자 특유의 빠른 판단과 실행 흐름은 유지한다.
- `/admin`, `/admin/dashboard`, `/admin/posts`, `/admin/profile`, `/admin/tools`의 역할과 카피를 추적 가능한 문서로 고정한다.

## 현재 코드 기준 대상
- 허브: `front/src/pages/admin.tsx`, `front/src/routes/Admin/AdminHubSurface.tsx`
- 운영 대시보드: `front/src/pages/admin/dashboard.tsx`
- 글 관리 작업실: `front/src/pages/admin/posts/index.tsx`, `front/src/routes/Admin/AdminPostsWorkspacePage.tsx`
- 프로필 작업실: `front/src/pages/admin/profile.tsx`
- 운영 도구 작업실: `front/src/pages/admin/tools.tsx`
- 공통 셸/프리미티브: `front/src/routes/Admin/AdminShell.tsx`, `front/src/routes/Admin/AdminSurfacePrimitives.tsx`

## 핵심 결정

### 1. 관리자 전체를 하나의 패턴으로 통일하지 않는다
- `허브`와 `운영 대시보드`는 공개 메인페이지 문법을 사용한다.
- `글 관리`, `프로필`, `운영 도구`는 공개 글 상세 페이지 문법을 사용한다.
- 이유:
  - 허브/대시보드는 `요약하고 판단하는 화면`이다.
  - posts/profile/tools는 `집중해서 수정하고 검토하는 화면`이다.
  - 현재 코드 구조도 이미 이 두 부류로 갈라져 있으므로, 레이아웃만 같은 언어로 정리하는 편이 가장 안전하다.

### 2. 브랜드 일관성은 시각 문법으로 맞춘다
- 메인/상세와 같은 수준의 타이포 대비를 사용한다.
- 큰 rounded slab 위주의 관리자 전용 콘솔 톤을 줄이고, borderless heading rail + 얇은 경계 + 여백 중심 surface를 기본으로 삼는다.
- 우측 rail은 계속 유지하되, 보조 정보만 맡긴다. 핵심 CTA를 rail로 보내지 않는다.

### 3. 카피는 메뉴명보다 행동과 상태를 먼저 말한다
- 관리자 제목은 `글 관리`, `프로필 설정`, `운영 도구`처럼 단순한 메뉴 라벨보다, 지금 이 화면에서 무엇을 해야 하는지 먼저 드러내야 한다.
- 설명 문장은 기능 소개보다 `왜 지금 이 화면을 봐야 하는지`를 먼저 설명한다.

## 공통 레이아웃 문법

### Admin shell
- 좌측 내비 + 상단 유틸 바 + 본문 캔버스 구조는 유지한다.
- 공통 shell은 관리자 전용 앱 셸이 아니라 `공개 페이지의 backstage extension`처럼 보여야 한다.
- 즉, 내비와 유틸 바는 기능적으로만 존재하고, 본문은 공개 페이지와 같은 리듬으로 읽혀야 한다.

### Hero
- 각 페이지 first fold는 `짧은 제목 1줄 + 설명 2줄 이내 + 우선 CTA 1~2개`로 제한한다.
- 큰 status slab, 중복 CTA, 메뉴명 반복은 두지 않는다.
- CTA는 `저장`, `새 글 작성`, `임시저장 이어쓰기`, `상태 새로고침`처럼 짧은 동사형으로 고정한다.

### Body
- 본문은 페이지당 핵심 목적 1개만 맡는다.
- 섹션 순서는 항상 `가장 먼저 봐야 하는 정보 -> 실제 작업 -> 보조 참고` 순으로 배치한다.
- KPI, 테이블, 편집 폼, 최근 작업은 모두 본문 안에 놓고, rail은 문맥 유지용으로만 쓴다.

### Rail
- rail의 책임:
  - 체크리스트
  - 최근 변경
  - 상태 의미
  - 빠른 이동
  - 런북/도움말
- rail에 두지 않을 것:
  - 1차 CTA
  - 메인 데이터 테이블
  - 긴 설명 블록

### 카피 규칙
- 제목은 명사형 메뉴명이 아니라 상태/행동 중심 문장으로 쓴다.
- 설명은 2문장 이하, 70자 안팎을 기본으로 한다.
- 버튼은 2~5글자 수준의 짧은 동사형으로 유지한다.

## 페이지별 패턴 고정

### `/admin`
- 패턴: 메인형 허브
- 역할: 하루 운영을 시작할 때 가장 먼저 보는 화면
- hero 제목:
  - `오늘 블로그 운영은 이 흐름으로 정리됩니다`
- hero 설명:
  - `새 글 작성, 최근 초안 복귀, 프로필 점검, 운영 상태 확인까지 지금 필요한 흐름만 먼저 보여줍니다.`
- hero CTA:
  - `새 글 작성`
  - `글 목록`
- 본문 섹션:
  1. `지금 할 일`
  2. `최근 작업`
  3. `공개 노출 상태`
- 우측 rail:
  - `프로필 완성도`
  - `빠른 이동`
  - `최근 변경`

### `/admin/dashboard`
- 패턴: 메인형 운영 대시보드
- 역할: 문제를 찾고 우선순위를 정하는 화면
- hero 제목:
  - `지금 확인해야 할 운영 상태`
- hero 설명:
  - `큐 지연, 메일 상태, 인증 이상, 파일 정리처럼 운영 리스크가 큰 항목부터 먼저 읽습니다.`
- hero CTA:
  - `상태 보기`
  - `운영 도구`
- 본문 섹션:
  1. `핵심 상태`
  2. `우선 점검`
  3. `시스템 패널`
- 우측 rail:
  - `최근 실패`
  - `런북`
  - `즉시 이동`

### `/admin/posts`
- 패턴: 글 상세형 작업실
- 역할: 목록 화면이 아니라 편집과 검수를 이어가는 작업실
- hero 제목:
  - `편집과 검수를 한 화면에서 이어갑니다`
- hero 설명:
  - `최근 초안 복귀, 공개 상태 점검, 목록 필터링까지 지금 필요한 글 작업 흐름을 한곳에 모읍니다.`
- hero CTA:
  - `새 글 작성`
  - `임시저장 이어쓰기`
- 본문 섹션:
  1. `지금 다룰 글`
  2. `최근 작업`
  3. `글 목록`
- 우측 rail:
  - `검수 체크리스트`
  - `상태 의미`
  - `바로가기`
- rail 보조 카피:
  - `공개 전에는 제목, 요약, 링크 노출 상태를 다시 확인하세요.`

### `/admin/profile`
- 패턴: 글 상세형 프로필 스튜디오
- 역할: 공개 메인과 About를 조율하는 작업실
- hero 제목:
  - `메인과 About에 보일 인상을 다듬습니다`
- hero 설명:
  - `프로필 이미지, 소개 문구, 헤더 카피, 외부 링크를 공개 화면 문맥에서 바로 조정합니다.`
- hero CTA:
  - `저장`
  - `미리보기`
- 본문 섹션:
  1. `프로필`
  2. `About 페이지`
  3. `헤더 문구`
  4. `외부 링크`
- 우측 rail:
  - `공개 노출 미리보기`
  - `빠진 항목`
  - `최근 저장`
- rail 보조 카피:
  - `메인 첫 화면과 About 상단에서 어떻게 읽히는지 먼저 확인하세요.`

### `/admin/tools`
- 패턴: 글 상세형 운영 리커버리 데스크
- 역할: 도구 모음이 아니라 장애 확인과 복구를 이어서 처리하는 작업실
- hero 제목:
  - `문제 확인과 복구를 같은 흐름에서 처리합니다`
- hero 설명:
  - `메일, 작업 큐, 정리 상태, 보안 이벤트처럼 장애와 직접 연결되는 항목만 우선 다룹니다.`
- hero CTA:
  - `상태 새로고침`
- 본문 섹션:
  1. `메일과 큐`
  2. `정리와 보안`
  3. `최근 진단 결과`
- 우측 rail:
  - `위험 액션`
  - `실행 전 체크`
  - `런북/장애 문서`
- rail 보조 카피:
  - `복구 액션은 현재 상태와 최근 실패 메시지를 확인한 뒤 실행합니다.`

## 카피 스타일 가이드
- 피해야 할 표현:
  - `관리 도구`
  - `설정`
  - `대시보드`
  - `목록 관리`
- 우선할 표현:
  - `지금 확인해야 할`
  - `이어갑니다`
  - `다듬습니다`
  - `처리합니다`
  - `작업실`
  - `운영 상태`
  - `공개 노출`

## 구현 시 주의점
- `AdminShell`은 유지하되, 본문 first fold의 인상은 공개 페이지와 맞춰야 한다.
- `AdminSurfacePrimitives`는 공통 표면과 rail 문법을 유지하는 기반으로 쓰고, page별 의미 색상만 얹는다.
- posts/profile/tools에서 rail은 sticky를 유지하되 CTA 과밀을 피한다.
- 허브/대시보드는 summary rail과 KPI를 유지하지만, 설명 문구를 과하게 늘리지 않는다.

## 완료 판단 기준
- 문서만 읽어도 어떤 관리자 화면이 메인형인지 상세형인지 바로 구분된다.
- 각 페이지의 hero 제목/설명/CTA와 본문 섹션 구조가 추적 가능하다.
- 공통 문법과 페이지별 차이가 한 문서 안에서 충돌 없이 설명된다.
